### 🔴 红线 8：判断视频生成完成的唯一标准 —— 检查 mediaUrl

- ✅ **判断逻辑**：API 返回数据中 `data.mediaUrl` **有值** = 视频已生成完成
- ✅ **立即行动**：检测到 `mediaUrl` 有值 → **立即下载视频** → **立即发送给用户**
- ❌ **严禁**检查 `data.status` 或 `data.repMsg.status` 字段来判断完成状态
- ❌ **严禁**看到 `status=1` 就认为还在处理中（`mediaUrl` 有值就是完成了！）

**案例教训（2026-03-31 任务 1733）**：
- API 返回 `data.status=1`（看似处理中），但 `data.mediaUrl` 已经有值了
- 我错误地认为还在处理中，没有立即下载发送
- 用户问"你走错接口了"、"是不是已经有视频地址了，你怎么没有发给我"才发现错误
- **正确做法**：只要 `mediaUrl` 有值，不管 `status` 是什么，都立即下载发送！

**正确轮询代码示例**：
```python
# ❌ 错误做法：检查 status
if data.get('status') == 2:
    download_video()

# ✅ 正确做法：检查 mediaUrl
if data.get('mediaUrl'):
    download_video(data['mediaUrl'])
```

## 📝 经验教训

**2026-03-30**：用户只发了一张图片生成视频，但我反复尝试多次都被系统中断或超时（任务一直显示 status=None 处理中），结果浪费了大量时间。其实第一次生成的任务虽然被中断，但 output 目录已有之前生成好的视频。

**教训**：
- ⚠️ 每次生成视频只能调用一次 workflow，调用前检查 output 目录是否已有待处理的视频
- ⚠️ 如果任务被中断，先检查 output 目录是否有已生成的视频，不要盲目重新调用

- ⚠️ aspectRatio 用驼峰写法（`--aspectRatio`，不是 `--aspect-ratio`）
- ⚠️ 不要使用 `--style` 和 `--quality` 参数
